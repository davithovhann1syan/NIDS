────────────────────────────────────────────────────────────────────────────────
  1. EXECUTIVE SUMMARY
────────────────────────────────────────────────────────────────────────────────

NIDS is a Python-based Network Intrusion Detection System designed for
low-to-medium traffic internal network segments (< 100 Mbps). It captures live
packets from a Linux network interface, runs them through a multi-stage rule
engine, deduplicates and correlates alerts, writes structured logs, sends
real-time notifications, and exposes a live web dashboard for monitoring and
investigation.

The system is purely signature-based — no machine learning or statistical
anomaly detection. Detection quality is determined entirely by the quality and
coverage of the signature rules. The current ruleset contains 74 rules spanning
11 threat categories.

Target environment: home or small office network running Linux.


────────────────────────────────────────────────────────────────────────────────
  2. SYSTEM ARCHITECTURE
────────────────────────────────────────────────────────────────────────────────

The system is structured as a two-thread pipeline with a third daemon thread
for outbound notifications and a fourth, fully isolated process for the web
dashboard.

  [Network Interface]
        │
        │  raw Scapy Packet objects
        ▼
  capture/sniffer.py          Thread 1 — Packet capture
        │  Scapy sniff() loop, BPF filter "ip", drop-on-full
        ▼
  capture/queue_manager.py    Shared state — bounded queue (10,000 packets)
        │  tracks enqueue count, drop count, drop rate
        ▼
  parser/extractor.py         Thread 2 — Analysis loop
        │  Scapy Packet → FeatureDict
        │  Fixed schema: timestamp, src_ip, dst_ip, protocol, length,
        │  ttl, src_port, dst_port, flags, flags_int, icmp_type
        ▼
  detection/sig_detector.py   Thread 2
        │  Evaluates all 74 rules: pattern, rate, multi_destination
        │  Returns list[Alert] — may be empty or multiple
        ▼
  detection/correlator.py     Thread 2
        │  Picks highest-severity alert as leader
        │  Annotates with also_triggered (all other fired rules)
        │  Computes cross-packet threat_score (0–100) per source IP
        │  Escalates severity on multi-stage attack patterns
        ▼
  alerting/deduplicator.py    Thread 2
        │  Suppresses (rule, src_ip) repeats within 30-second cooldown
        ▼
  alerting/logger.py          Thread 2
        │  Writes one JSON line per alert to logs/nids.log
        │  Line-buffered; supports SIGHUP log rotation reopen
        ▼
  alerting/notifier.py        Thread 3 (daemon) — Notifications
        │  Batches alerts within a 3-second window
        │  Sends email (SMTP) and/or Slack webhook for HIGH/CRITICAL
        ▼
  [logs/nids.log]             Append-only JSON lines on disk

  dashboard/app.py            Separate process — Web dashboard
        Reads nids.log only — fully isolated from the detection layer
        Serves live and historical alert data via a Flask REST API

Design principles:
  - One-way data flow: no module imports anything "above" it in the pipeline.
  - Shared state is minimized: only the bounded queue crosses thread boundaries.
  - The dashboard has zero runtime coupling to the detection layer.
  - Credentials (SMTP, Slack) are loaded from environment variables only.

                Used for the timeline line chart, severity doughnut, horizontal

                Allows credentials to be stored outside source control.

                Used by the notifier for email delivery. Maintains a
                persistent connection and reconnects automatically on

                Used by the notifier to POST alerts to a Slack
                incoming webhook URL.

                Thread 1: sniffer. Thread 2: analysis loop.
                Thread 3: notifier daemon.

                Decouples sniffer from analysis. Bounded at 10,000 packets;

collections     Standard library — defaultdict, deque.
                Sliding-window state for rate and multi_destination rules
                is stored as deques keyed by (src_ip, rule_name).

typing          Standard library — TypedDict, Literal, NotRequired.
                All inter-module data structures (FeatureDict, Alert,
                CorrelatedAlert, Rule subtypes) are fully typed.


────────────────────────────────────────────────────────────────────────────────
  4. CORE FEATURES
────────────────────────────────────────────────────────────────────────────────

  4.1  PACKET CAPTURE
  ─────────────────────────────────────────────────────────────────────────────
  - Live capture via Scapy sniff() on any named Linux network interface.
  - BPF filter "ip" applied at the kernel level — only IPv4 packets reach
    Python, reducing userspace overhead on busy interfaces.
  - Bounded queue with configurable max size (default: 10,000 packets).
    Overflow drops the arriving packet rather than blocking the sniffer,
    preventing backpressure from stalling the capture thread.
  - Drop rate tracked in real time: drop_count / enqueue_count × 100 %.
  - Graceful shutdown: Ctrl+C sets a stop_event, the sniffer thread exits,
    and the analysis loop drains any remaining packets before closing the
    log file.

  4.2  FEATURE EXTRACTION
  ─────────────────────────────────────────────────────────────────────────────
  - Converts a raw Scapy Packet object into a flat FeatureDict with a
    fixed, always-complete schema:
      timestamp, src_ip, dst_ip, protocol (int), length, ttl,
      src_port, dst_port, flags (string), flags_int (int), icmp_type
  - No key is ever absent — missing fields default to None or 0 so
    condition matching never raises a KeyError.
  - Raises ValueError for non-IP packets that bypass the BPF filter;
    these are silently discarded upstream.

  4.3  SIGNATURE DETECTION ENGINE
  ─────────────────────────────────────────────────────────────────────────────
  Three rule types:

  pattern         Fires on a single packet that satisfies all conditions.
                  Stateless — no per-IP memory required.
                  Example: Null Scan (TCP with flags_int == 0).

  rate            Fires when a single source IP sends ≥ threshold matching
                  packets within window_seconds.
                  State: sliding-window deque per (src_ip, rule_name).
                  Example: SSH Brute Force (≥ 10 SYN packets to port 22
                  within 10 seconds).

  multi_destination  Fires when a single source IP contacts ≥ threshold
                  unique values of a tracked field (dst_port or dst_ip)
                  within window_seconds.
                  More accurate than raw packet counts for scan detection —
                  a client opening 50 connections to one port scores 1,
                  not 50.
                  Example: Port Scan (Distinct Ports) tracks unique dst_port
                  values.

  Condition matching supports five forms per field:
    Equality        "protocol": 6
    Membership      "dst_port": [80, 443, 8080]
    Comparison ops  {">=": 1400}  — also >, <, <=, !=
    Exclusion       {"not_in": [22, 80]}
    Bitmask         {"mask": 0x02}  — all bits must be set

  Periodic purge: stale sliding-window entries for quiet source IPs are
  evicted every 2 minutes to prevent unbounded memory growth during scans.

  4.4  ALERT CORRELATION
  ─────────────────────────────────────────────────────────────────────────────
  Per-packet correlation:
  - All rules that fire on the same packet are collected.
  - The rule with the highest severity becomes the "leader" of the alert.
  - All other fired rule names are attached as also_triggered, giving the
    analyst the full picture in a single log entry.

  Cross-packet threat tracking:
  - The correlator maintains a 5-minute sliding window of distinct rules
    fired per source IP.
  - A threat_score (0–100) is computed: 10 points per distinct rule fired.
  - Severity escalation on multi-stage attack patterns:
      3+ distinct rules  (score ≥ 30) → escalate to at least MEDIUM
      5+ distinct rules  (score ≥ 50) → escalate to at least HIGH
      8+ distinct rules  (score ≥ 80) → escalate to CRITICAL
  - An IP that scans, then brute-forces, then beacons to a C2 port will
    have its alerts automatically escalated to CRITICAL, even if each
    individual rule fires at MEDIUM.

  MITRE ATT&CK mapping:
  - Most rules carry a mitre field (e.g. "T1046" for network service
    scanning) for correlation with the ATT&CK framework.

  4.5  DEDUPLICATION
  ─────────────────────────────────────────────────────────────────────────────
  - Each (rule, src_ip) pair has an independent 30-second cooldown timer.
  - Only the first alert in each window is passed to the logger and
    notifier; subsequent duplicates are silently counted.
  - suppressed_count is exposed on the Deduplicator instance for stats.
  - Expired cooldown entries are purged every 2 minutes alongside the
    detector's rate state.

  4.6  LOGGING
  ─────────────────────────────────────────────────────────────────────────────
  - Every non-duplicate alert is written as one JSON line to logs/nids.log.
  - Fields per line: timestamp (ISO 8601 UTC), rule, severity, src_ip,
    dst_ip, dst_port, protocol, correlated, also_triggered, threat_score,
    and count (rate/multi_destination rules only).
  - Line-buffered: each line is flushed to disk immediately, so no alert
    is lost if the process crashes.
  - SIGHUP reopens the log file without restarting, compatible with the
    standard logrotate copytruncate / postrotate workflow.

  4.7  NOTIFICATIONS
  ─────────────────────────────────────────────────────────────────────────────
  - A dedicated daemon thread handles all outbound I/O so the analysis
    loop is never blocked by network latency.
  - Configurable minimum severity (default: HIGH). LOW and MEDIUM alerts
    are logged but not notified.
  - Alert batching: alerts arriving within a 3-second window are collapsed
    into a single notification, preventing email storms during active attacks.
  - Bounded internal queue (100 slots): when full, the oldest pending alert
    is evicted. An analyst reading email hours later sees the most recent
    events, not the first ones from a long-over incident.
  - Email (SMTP/STARTTLS): persistent connection reused across sends with
    one automatic reconnect attempt on stale-socket failures.
  - Slack (incoming webhook): formatted with severity-appropriate emoji,
    multi-alert batches sent as a single message.
  - All credentials loaded from environment variables; never hardcoded.

  4.8  WEB DASHBOARD
  ─────────────────────────────────────────────────────────────────────────────

  - Two independent Flask applications, both reading detection outputs
    directly from disk — neither imports anything from the detection layer.

  5.1  DETECTION REFACTOR — FALSE POSITIVE REDUCTION
  ─────────────────────────────────────────────────────────────────────────────

  The original ruleset had 83 rules. Nine were removed and several were
  converted to less noisy rule types to fix three real problems observed
  during testing:

  Problem 1 — Own IP appearing as "top attacker"
    Cause:  Outbound packets from the sensor machine have src_ip = local IP.
            Rate rules accumulated hits → threat_score escalated → local IP
            flagged CRITICAL.
    Fix:    main.py now auto-detects the interface IP via get_if_addr() and
            skips any packet whose src_ip is in {local_ip, 127.0.0.1, ::1}.

  Problem 2 — Kubernetes/random alerts during nmap scans
    Cause:  All 9 Exposed Services rules were "pattern" type — one SYN to
            port 6443 fired "Kubernetes API Exposed". nmap probes each port
            once → 9 distinct rules → correlator escalated to CRITICAL.
    Fix:    All Exposed Services rules (R048–R056) converted from pattern →
            rate with threshold=3/60s. nmap sends exactly 1 SYN per port and
            never reaches 3, so no alert fires.

  Problem 3 — False positives and inconsistent alerts
    Nine noisy pattern rules removed:
      SMB Access       — every Windows share mount triggered it
      DCOM/RPC         — fires on every Windows background event
      NetBIOS          — fires on every Windows name query
      LDAP Enumeration — constant in Active Directory environments
      SNMP Access      — every monitoring tool poll
      NFS Access       — every NAS mount
      SSDP/UPnP        — constant on home/office LANs
      FTP Cleartext    — every FTP packet; R019 covers brute-force attacks
      Log4Shell Target — port 8080/8443 too common; port-based detection
                         is useless for Log4Shell

    Three rules converted pattern/rate → multi_destination:
      R007 Maimon Scan  (FA to 20 unique dst_ports/30s — was: 20 FA pkts)
      R008 ACK Scan     (A to 30 unique dst_ports/30s  — was: 30 ACK pkts)
      R039 Reverse Shell (SYN to 5 unique ports>60000/60s — was: single SYN)
    Key insight: scanners probe many different ports; legitimate connections
    reuse the same small port set. Unique-count tracking eliminates the
    false positive entirely.

    Other threshold/condition fixes:
      Port Scan (SYN): raised 30/5s → 100/10s (browser opening 30 tabs
                        was triggering the old threshold)
      Telnet Attempt:  added flags="S" (SYN only) to suppress established
                        session packet noise
      C2 port list:    removed 8888 (common Jupyter port) and 9000 (used by
                        Portainer, SonarQube, PHP-FPM)

    nids --interface wlan0

    sudo .venv/bin/python scripts/test_capture.py

    # Traffic generator (root required) — 52 attacks, 10 scenarios

────────────────────────────────────────────────────────────────────────────────
  7. KNOWN LIMITATIONS
────────────────────────────────────────────────────────────────────────────────

  - IPv4 only. The BPF filter "ip" excludes IPv6 traffic entirely. An
    attacker on a dual-stack network can evade detection by using IPv6.

  - Single interface. The sniffer binds to one interface per process. A
    multi-homed host requires running multiple instances.

  - Rule-based only. There is no behavioral baseline or statistical anomaly
    detection. Novel attacks that do not match any signature are invisible.

  - Signature false positives. Rate rules (e.g. Port Scan (SYN)) can fire
    on legitimate high-traffic clients. Thresholds are tuned conservatively
    but will need adjustment for busy environments.

  - No rule hot-reload. Adding or editing signatures requires a process
    restart.

  - Dashboard has no authentication. Anyone with network access to port 5000
    can read alert data. Running it on a loopback or behind a firewall is
    strongly advised.

  - No HTTPS. Dashboard traffic is unencrypted.

  8. FUTURE IMPROVEMENTS
────────────────────────────────────────────────────────────────────────────────

  NEAR-TERM (operational gaps)

  [x] .env.example — committed at root, documents every required variable.

  [x] systemd service files — deploy/nids.service (capture, root) and
        deploy/nids-dashboard.service (web UI, normal user).

  [x] logrotate configuration — deploy/nids-logrotate, daily/30-day/SIGHUP.

  [x] IP allowlist — allowlist.py + allowlist.json; supports IP/CIDR entries,
        editable from the dashboard, hot-reloaded every 2 minutes.

  [x] Full-featured web dashboard — 7 views (Dashboard, Alert Feed, Attack Map,
        Rules, Analytics, Allowlist, PCAP Analysis), collapsible sidebar,
        dark/light theme, IP investigation drawer, toast notifications,
        keyboard shortcuts, CSV export, capture engine control, demo mode.

  [x] Offline PCAP analysis — drag-and-drop upload in both the standalone
        pcap_app.py server and as an embedded tab in the live dashboard.

  [ ] Notifier rate-limit
        The notifier already batches alerts within a 3-second window,
        but a sustained multi-hour attack can still produce hundreds of
        batched emails. Add a maximum notifications-per-hour cap so the
        analyst's inbox does not flood.

  MEDIUM-TERM (capability improvements)

  [ ] IPv6 support
        Change the BPF filter from "ip" to "ip or ip6" and extend
        extractor.py to handle IPv6 headers. The signatures would then
        also work for IPv6 flows.

  [ ] Multi-interface support
        Allow a single NIDS process to sniff multiple interfaces
        simultaneously (e.g. wlan0 and eth0) by running one sniffer
        thread per interface, all writing to the same bounded queue.

  [ ] GeoIP enrichment
        Annotate alerts with the country code and ASN of the source IP
        using a local MaxMind GeoLite2 database. Adds context to the
        dashboard without requiring an external API call per packet.

  [ ] SIEM integration (syslog / CEF output)
        Write alerts to a syslog socket or in CEF (Common Event Format)
        in addition to JSON lines. Enables forwarding to Splunk, ELK,
        or any syslog-capable SIEM without post-processing.

  [ ] Dashboard authentication
        Add HTTP Basic Auth or token-based authentication to the Flask
        app so alert data is not readable by anyone on the local network.
        Even a single shared password behind python-dotenv would be a
        meaningful improvement.

  [ ] HTTPS for dashboard
        Wrap Flask with a self-signed certificate (or use a reverse
        proxy such as Nginx with Let's Encrypt) so dashboard traffic
        is encrypted.

  [ ] Rule hot-reload
        Add a SIGUSR1 handler that reloads detection/signatures.py at
        runtime without stopping packet capture. Would allow adding new
        rules to a running deployment without a service restart.

  [x] Alert CSV export
        Dashboard "Alert Feed" view has a CSV export button that downloads
        the current filtered view. PCAP analysis results are also exportable.

  LONG-TERM (architectural additions)

  [ ] Anomaly detection layer
        Complement the signature engine with a lightweight statistical
        baseline (e.g. exponential moving average of packet rate per
        source IP, or port entropy per time window). Anomaly alerts
        would surface novel attack patterns that no rule currently covers,
        at the cost of a higher false-positive rate that would need tuning.

  [ ] Machine learning classifier
        Train a binary classifier (e.g. Random Forest or a small MLP)
        on labeled FeatureDicts to flag suspicious packets that bypass
        all signature rules. Requires a labeled pcap dataset and adds a
        dependency on scikit-learn or PyTorch.

  [ ] Packet capture to PCAP
        Optionally write raw captured packets to a rotating PCAP file
        alongside the JSON alert log. Allows full forensic replay of an
        incident after the fact.

  [ ] Distributed mode
        Support multiple sensor nodes each running the capture and
        detection pipeline, forwarding correlated alerts to a central
        aggregator that runs the dashboard. Useful for monitoring
        multiple network segments.

  [ ] Custom rule editor in dashboard
        A form in the web UI to define, test, and activate new signatures
        without editing Python files. Rules would be serialized to a
        user-rules.json file and merged with the built-in SIGNATURES list
        on startup.

  [ ] MITRE ATT&CK coverage report
        Generate a heatmap overlay on the ATT&CK Navigator matrix showing
        which techniques the current ruleset covers, and which are blind
        spots. Useful for identifying gaps before a security assessment.
